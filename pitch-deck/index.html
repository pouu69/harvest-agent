§ 01 / THE PROBLEM

매번 휘발된다.

Claude Code 세션 한 번에 만들어진 가치 있는 지식이,
다음 세션으로 단 한 줄도 넘어가지 않는다.

  • D Decisions 왜 이 라이브러리를 골랐는가, 왜 이 아키텍처인가
  • L Learnings 새로 알게 된 패턴 · 트릭 · 도구 사용법
  • R Reusable 다른 곳에서도 쓸 만한 스니펫과 접근법
  • A Anti-patterns 시행착오로 알게 된 함정 — 가장 비싼 손실

결과 → 같은 실수 반복, 이미 내린 결정의 재논의.

§ 02 / THE SOLUTION

두 개의 명령. 그리고 마크다운.

i.
$ harvest init

프로젝트에 .harvest/ 폴더(=KB) 생성
+ CLAUDE.md 마커 블록에 @-import 한 줄

──▶
ii.
며칠간 Claude Code 사용

평소대로 작업하면 ~/.claude/projects/ 아래에 transcript가 쌓인다.

──▶
iii.
$ harvest start

미처리 transcript를 분석. Agent가 항목을
create · merge · supersede · archive.

~/projects/my-blog 
$ harvest start
✓ 7 sessions → 3 processed, 4 trivial-skipped
   Created:  D-012 (auth-jwt-strategy)
             A-001 (jwt-refresh-loop)
   Merged:   L-005
   Cost:     $0.62  ·  4m 18s

다음 Claude Code 세션은 INDEX 를 자동 로드 → 같은 함정을 두 번 밟지 않는다.

§ 03 / TAXONOMY

네 개의 카테고리. 고정.

자유 폼 태그가 아니라 의견이 있는 라우팅.
각 항목은 정확히 하나의 폴더에 들어간다.

DDecisions

선택의 왜. 라이브러리, 아키텍처, 트레이드오프의 근거.

.harvest/decisions/D-XXX-*.md
LLearnings

새로 발견한 도구, 패턴, 동작 — 다음 번에 빠르게 도달하기 위해.

.harvest/learnings/L-XXX-*.md
RReusable

다른 자리에 그대로 옮길 수 있는 스니펫과 접근법.

.harvest/reusable/R-XXX-*.md
AAnti-patternsCRITICAL

반복하면 안 되는 함정. severity:critical은 INDEX 최상단에 항상 노출 — 다음 세션이 즉시 본다.

.harvest/anti-patterns/A-XXX-*.md
§ 04 / DIFFERENTIATOR I

Push, not Pull.

RAG · LLM-WIKI

사용자가 묻는다.
그제서야 검색이 일어나고,
그제서야 답이 만들어진다.

PULL · on-demand · query-shaped

vs
HARVEST

세션이 시작되는 그 순간,
INDEX가 컨텍스트에 이미 들어가 있다.
CLAUDE.md @-import 한 줄.

PUSH · ambient · index-shaped

Claude는 묻지 않아도 알고 시작한다. 검색 단계가 통째로 사라진다.

§ 05 / DIFFERENTIATOR II

Concentration, not Storage.

per category
decisions ≤ 10 · learnings ≤ 10 ·
reusable ≤ 10 · anti-patterns ≤ 10
×
40
per KB
한 지식 베이스의 총 항목 수.
넘치면 자율 정리가 작동.

신규 작성은 마지막 수단. 우선순위는:

  1. 01merge기존 항목과 병합 — 중복 제거 + 사례 누적
  2. 02supersede새 결정이 옛 결정을 대체 — History 보존
  3. 03archive관련성을 잃은 항목을 인덱스에서 내림
  4. 04create위 셋이 모두 적합하지 않을 때만

노이즈가 시그널을 죽이지 않도록, 의도적으로 작게 유지한다.

§ 08 / INVARIANTS

중단해도, 두 번 돌려도, 안전하다.

⊕

Idempotent

processed.json ledger가 (session_id, sha256)를 키로 중복 처리를 차단. 두 번 돌려도 같은 결과.

⌬

Atomic write

모든 KB 파일 쓰기는 temp + rename. 부분 실패가 KB를 깨뜨릴 수 없음.

▣

KB-단위 lock

체인의 모든 KB lock을 all-or-nothing으로 획득. 충돌 시 즉시 exit 4.

↺

2-stage SIGINT

1차 Ctrl+C → AI SDK 호출 cooperative abort + INDEX 재빌드 + lock 해제 → exit 130. 2차 → sync cleanup.

네 invariants 모두 코드의 계약 — 새 코드는 이걸 깨면 lint/typecheck 단계에서 막힌다.

§ 10 / SCALE

모노레포가 자라도, 지식은 새지 않는다.

my-monorepo/
├── .harvest/                ← root KB · 보편 지식
│   └── INDEX.md
│
├── apps/
│   ├── web/
│   │   ├── src/
│   │   └── .harvest/        ← web 영역 KB
│   │
│   └── api/
│       ├── src/
│       └── .harvest/        ← api 영역 KB
│
└── packages/
    └── shared-utils/
        └── .harvest/        ← 공유 유틸 KB
→

가장 가까운 KB가 우선. 자식 KB의 가이드가 root와 충돌하면 자식이 이긴다.

↑

자동 promotion. 같은 패턴이 둘 이상의 자식 KB에서 발견되면 root로 승격.

⊥

격리. app 전용 지식이 root로 새지 않도록 영역 마스킹이 강제된다.

§ 11 / RECAP · 한 페이지 요약

휘발성 세션을 영속 KB로.

Claude Code 세션이 매번 같은 실수를 반복하지 않도록,
다섯 가지만 기억하면 됩니다.

  1. 01 Problem 세션은 매번 휘발된다 — Decisions · Learnings · Reusable · Anti-patterns가 다음 세션으로 한 줄도 넘어가지 않는다.
  2. 02 Solution harvest init → harvest start. 두 명령으로 transcript를 4-카테고리 마크다운 KB로 수확하고, CLAUDE.md @-import로 다음 세션에 자동 주입.
  3. 03 Position Push, not Pull. Concentration, not Storage. 카테고리당 10·KB당 40 cap, 머지·supersede·archive가 신규 작성보다 우선.
  4. 04 Architecture 결정론 우선 — LLM은 EXTRACT + tie-break 두 곳에서만. atomic write · KB-단위 lock · idempotent ledger · 2-stage SIGINT.
  5. 05 Surface 3 providers (Anthropic / OpenAI / Google) · 모노레포 KB chain · 평문 마크다운 + git. 두 개의 명령. 그 외 표면 없음.
